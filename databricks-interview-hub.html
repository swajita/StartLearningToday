Fundamentals

Core concepts interviewers often ask before they move into Spark internals, optimization, or architecture trade-offs.

Starter layer

What is Databricks, and how is it different from a traditional data platform?

PlatformLakehouseBeginner
+

Databricks is a unified analytics platform built around the lakehouse model, so data engineering, analytics, machine learning, and governance can work on the same foundation instead of separate tools.

A traditional stack often splits storage, ETL, BI, and ML across multiple systems, while Databricks tries to reduce movement, duplication, and operational overhead by centralizing compute and data workflows.

What is the lakehouse architecture?

ArchitectureConcept
+

A lakehouse combines the low-cost, flexible storage style of a data lake with the management and performance features people expect from a warehouse.

The key point is that you can keep raw and curated data in open storage formats while still getting ACID transactions, schema enforcement, and faster analytics with technologies such as Delta Lake.

What are the main components inside a Databricks workspace?

WorkspaceNavigation
+

Common components include notebooks, clusters or compute, jobs, workflows, repos, SQL warehouses, data catalogs, and model or serving features depending on the edition.

A good interview answer also mentions that workspaces support collaboration across personas such as analysts, data engineers, and ML practitioners.

What is the difference between managed and external tables?

TablesStorage
+

Managed tables are controlled by the platform, so Databricks manages both metadata and underlying data lifecycle.

External tables keep the data in a user-managed location, while the metastore points to it; this is useful when teams want tighter control over storage paths, sharing, or lifecycle policies.

Apache Spark

Questions here test whether you understand execution, partitioning, optimization, and how Databricks uses Spark under the hood.

Execution layer

What is lazy evaluation in Spark, and why does it matter?

SparkExecution
+

Lazy evaluation means Spark does not execute transformations immediately; instead, it builds a logical plan and waits until an action is triggered.

This matters because Spark can optimize the execution plan, reduce unnecessary work, and combine stages more efficiently before actual computation starts.

What is the difference between repartition and coalesce?

PartitionsPerformance
+

Repartition reshuffles data and can both increase or decrease partitions, which makes it flexible but more expensive.

Coalesce usually reduces partitions with less shuffle, so it is often preferred when you want to shrink partition count after filtering or before writing output.

How do narrow and wide transformations differ?

StagesShuffle
+

Narrow transformations do not require data movement across partitions, so each output partition depends on a small number of input partitions.

Wide transformations need a shuffle, such as in groupBy or join operations, because data must move across executors to regroup records.

When would you use a broadcast join?

JoinsOptimization
+

You use a broadcast join when one side of the join is small enough to send to each executor, which avoids a large shuffle on both sides.

This can improve performance significantly, but you still need to consider memory limits and avoid broadcasting data that is larger than expected.

Delta Lake

These questions usually appear in Databricks-focused rounds because Delta is central to reliability, versioning, and performance.

Storage layer

How does Delta Lake provide ACID transactions on top of cloud object storage?

ACIDTransaction Log
+

Delta Lake keeps a transaction log that records table changes in an ordered, versioned way, which lets readers and writers coordinate reliably.

Because the log tracks commits, snapshots, schema changes, and file-level operations, the table behaves more like a database table than raw files in storage.

What is time travel in Delta Lake?

VersioningRecovery
+

Time travel lets you query or restore older versions of a Delta table using version numbers or timestamps.

It is helpful for debugging bad data loads, reproducing reports, and rolling back accidental changes during development or operations.

What do OPTIMIZE, Z-ORDER, and VACUUM do?

MaintenancePerformance
+
  • OPTIMIZE compacts many small files into fewer larger files for better read efficiency.
  • Z-ORDER colocates related values to improve data skipping for common filter columns.
  • VACUUM removes obsolete files that are no longer needed after retention requirements are met.

What is the difference between schema enforcement and schema evolution?

GovernanceSchema
+

Schema enforcement prevents invalid or unexpected writes when incoming data does not match the target table definition.

Schema evolution allows controlled updates to the schema, such as adding new columns, when the change is intentional and supported by the write operation.

SQL & Warehousing

Useful for analytics engineering, BI, and platform roles that mix Databricks SQL with notebooks and jobs.

Analytics layer

What is a Databricks SQL warehouse?

Databricks SQLWarehouse
+

A Databricks SQL warehouse is compute optimized for SQL queries, dashboards, and BI workloads instead of notebook-first engineering tasks.

In interviews, it helps to mention that warehouses separate SQL consumption from general-purpose Spark clusters, which improves governance and workload management.

How are CTEs, views, temp views, and global temp views different?

SQLObjects
+

A CTE exists only inside a query, a regular view stores reusable query logic, a temp view is session-scoped, and a global temp view can be shared across sessions in the same application context.

This question checks whether you understand scope, persistence, and where each object is useful.

What techniques help optimize SQL queries in Databricks?

OptimizationSQL Tuning
+
  • Use partitioning and clustering strategies that match common filters.
  • Reduce small files and keep statistics fresh.
  • Leverage predicate pushdown, column pruning, and efficient join strategies.
  • Review the physical plan instead of guessing.

Why are window functions common in Databricks SQL interviews?

Window FunctionsAnalytics
+

Window functions solve ranking, running totals, lead-lag, and de-duplication problems without collapsing rows like aggregate queries do.

Interviewers use them because they reveal whether you can express business logic clearly and efficiently in SQL.

Architecture

These questions assess whether you can design reliable pipelines and reason about cost, scalability, and governance.

System design

What is the medallion architecture in Databricks?

BronzeSilverGold
+

The medallion pattern organizes data into bronze, silver, and gold layers to separate raw ingestion, cleaned transformation, and business-ready consumption.

This improves traceability, data quality, and reusability because each layer has a clear purpose and contract.

How would you choose between batch and streaming pipelines?

StreamingTrade-offs
+

The choice depends on latency needs, source behavior, downstream expectations, operational complexity, and cost.

If the business needs near-real-time updates and can support checkpointing, late-arriving data handling, and monitoring, streaming may be justified; otherwise, well-designed batch pipelines are often simpler and cheaper.

What problem does Unity Catalog solve?

GovernanceSecurity
+

Unity Catalog centralizes governance for data, AI assets, permissions, lineage, and discovery across workspaces.

A strong answer highlights that centralized access control and auditability become easier when governance is not spread across isolated metastores or teams.

When should you use job clusters versus all-purpose clusters?

ComputeCost Control
+

Job clusters are typically better for scheduled or automated workloads because they are ephemeral, isolated, and easier to align with specific jobs.

All-purpose clusters are more useful for interactive development, ad hoc exploration, and shared notebook work, but they need tighter cost and governance controls.

Scenario Round

Practical prompts that push you beyond definitions and into design choices, troubleshooting, and stakeholder communication.

Applied thinking

A pipeline suddenly becomes slow after a new release. How would you investigate?

TroubleshootingScenario
+

Start by comparing runs before and after the release, checking execution plans, shuffle behavior, input volumes, small-file growth, cluster changes, and skew indicators.

Then isolate whether the issue is code logic, data characteristics, cluster sizing, or storage layout, and propose fixes with measurable evidence rather than assumptions.

How would you handle late-arriving data in a streaming solution?

Structured StreamingData Quality
+

You would usually define watermarking, deduplication logic, and acceptable lateness based on the business SLA instead of purely technical defaults.

It is important to explain the trade-off: stricter lateness windows reduce state and cost, but they may drop legitimate delayed records.

How would you migrate an existing on-prem warehouse workload to Databricks?

MigrationPlanning
+

Break the migration into assessment, prioritization, data movement, query conversion, validation, performance tuning, and operating-model setup.

Interviewers usually want to hear that you would migrate in waves, prove correctness with parallel runs, and redesign where needed instead of blindly lifting every legacy pattern.

How would you respond if a stakeholder asks for real-time dashboards but the source refreshes hourly?

CommunicationRequirements
+

Clarify the business meaning of real time first, because many requests are actually about faster visibility rather than true second-level freshness.

A mature answer balances expectations, source constraints, cost, and business value, then proposes the simplest architecture that meets the real need.
